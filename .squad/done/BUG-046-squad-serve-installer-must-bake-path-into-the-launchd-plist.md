## Problem

The launchd plist generated by `internal/tui/daemon/install_darwin.go`
sets `EnvironmentVariables` with `SQUAD_HOME` only — no `PATH`. The
launchd-spawned daemon therefore inherits launchd's default PATH
(`/usr/bin:/bin:/usr/sbin:/sbin`), which does NOT include
`/opt/homebrew/bin` (Apple-Silicon Homebrew), `/usr/local/bin` (Intel
Homebrew), `~/.claude/local`, or `~/go/bin`. The auto-refine handler
(`internal/server/items_auto_refine.go:48`) calls
`exec.LookPath("claude")` and returns 503 with body
`{"error": "claude CLI not found on PATH"}` even when `claude` is
trivially reachable from the user's interactive shell. Reproduced
today: `which claude` returned `/opt/homebrew/bin/claude`; the
daemon's `ps -E` showed `PATH=/usr/bin:/bin:/usr/sbin:/sbin`; the SPA
auto-refine click 503'd. Same shape applies to `git` and any other
external binary squad shells out to.

## Context

The systemd-user service file
(`internal/tui/daemon/install_linux.go`) has the same gap — only
`Environment=SQUAD_HOME=...` is set; no `PATH`. systemd's default
PATH is `/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin`,
so it covers Intel-Homebrew and many distro packages, but not Apple
Silicon Homebrew (irrelevant on linux) or `~/.claude/local`. Worth
fixing here too for consistency and for distros that set a narrower
default.

The runtime workaround verified during BUG-040 close was
`launchctl setenv PATH ...` followed by `launchctl kickstart -k`.
That works for the current launchd session but is lost on reboot.

The right fix is in the plist / unit-file template: add `PATH` to
`EnvironmentVariables` (darwin) / `Environment` (linux) with a
generous default that covers Homebrew-on-arm64, Homebrew-on-amd64,
the system dirs, the Anthropic curl-script install location
(`~/.claude/local`), and the Go install location (`~/go/bin`).

## Acceptance criteria

- [ ] The darwin plist template emits a `PATH` entry under `EnvironmentVariables` that includes `/opt/homebrew/bin`, `/usr/local/bin`, `/usr/bin`, `/bin`, `/usr/sbin`, `/sbin`, `<HomeDir>/.claude/local`, and `<HomeDir>/go/bin`.
- [ ] The linux unit-file template emits an `Environment=PATH=...` line with the equivalent set (omit the macOS-only Homebrew arm64 prefix; include `/usr/local/bin`, `/usr/bin`, `/bin`, `/usr/local/sbin`, `/usr/sbin`, `/sbin`, `<HomeDir>/.local/bin`, `<HomeDir>/.claude/local`, `<HomeDir>/go/bin`).
- [ ] After `squad serve --reinstall-service` and `launchctl kickstart -k gui/$UID/sh.squad.serve` (or the systemd-user equivalent), the daemon's runtime `PATH` includes the new entries — verifiable via `ps -E -p <pid> | tr ' ' '\n' | grep ^PATH=`.
- [ ] Existing install tests in `install_darwin_test.go` and `install_linux_test.go` continue to pass; new assertions confirm the rendered plist / unit body contains the expected PATH substrings.

## Notes

Resist the temptation to capture the user's interactive shell PATH at
install time. That's tempting but brittle — the user's PATH could
include project-local node_modules/.bin, virtualenvs, or other paths
that go stale and confuse the daemon long-term. A hardcoded
"reasonable defaults" PATH is more predictable and matches how
Homebrew's own launchd-managed services handle this.

If a future user truly needs a custom PATH (say, NixOS or
non-standard Homebrew prefix), the workaround is to edit the plist
manually — same escape hatch every other PATH-baking installer uses.
